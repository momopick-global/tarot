# 문서 관리 가이드 (YourTarot)

프로젝트 문서를 "읽기 쉽고, 최신 상태로, 유지 가능한 방식"으로 관리하기 위한 운영 기준입니다.

---

## 1) 관리 목표

- **일관성:** 문서 톤/형식/용어를 하나의 기준으로 맞춘다.
- **최신성:** 코드/DB/API와 문서가 최대한 동일한 상태를 유지한다.
- **접근성:** 새로 합류한 사람도 README와 docs 인덱스만 보고 빠르게 시작할 수 있다.
- **지속성:** 문서 관리를 개인 습관이 아닌 팀 프로세스로 운영한다.

---

## 2) 문서 작업 유형 (우선순위 기준)

### A. 전체 문서 톤/형식 통일

**목적:** 문서를 읽기 쉽게 "한 스타일"로 통일  
**작업:** 제목 규칙, 섹션 순서, 용어/문체 통일, 목차 추가  
**특징:** 내용 자체보다 문서 품질(가독성/일관성) 개선 중심

### B. README 중심 최신화

**목적:** 새로 온 사람이 `README.md` 하나만 보고 시작 가능하도록 구성  
**작업:** 설치/실행, 환경변수, 빌드/배포, 자주 쓰는 명령 업데이트  
**특징:** 온보딩/사용성 중심, 영향 파일이 상대적으로 적음

### C. `docs/` 내부 기술문서 최신화

**목적:** 실제 코드/DB/API 상태와 기술 문서를 일치시킴  
**작업:** `docs/api-spec.md`, `docs/data-model.md`, 체크리스트 문서 반영  
**특징:** 정확성/운영 안정성 중심, 팀 개발에서 실무 효과가 큼

### D. 특정 파일만 지정 수정

**목적:** 필요한 문서만 빠르게 수정  
**작업:** 지정 파일 + 지정 항목만 수정  
**특징:** 가장 빠르고 비용이 적지만, 전체 일관성은 덜 챙길 수 있음

---

## 3) 추천 실행 순서 (가성비 기준)

1. **1단계: B + D**  
   - 체감 개선이 빠르고 리스크가 낮음
2. **2단계: C**  
   - 실서비스/운영 기준에서 정확도 확보
3. **3단계: A**  
   - 문서 경험 품질 마무리
4. **상시 운영: 아래 5장 프로세스 적용**

---

## 4) 문서 스타일 기준

### 4-1. 제목/구조 규칙

- 문서 제목은 `# 문서명 (프로젝트명)` 형식 권장
- 주요 구역은 `##`, 세부는 `###`까지 사용 (과도한 깊이 금지)
- 문서 상단에 목적 1~2문장 기재
- 긴 문서는 "목차" 또는 "빠른 이동 섹션" 제공

### 4-2. 문체/용어 규칙

- 문체는 간결한 서술형으로 통일
- 같은 개념은 같은 단어로만 표기 (예: "마이페이지" vs "내 페이지" 혼용 금지)
- 실행 가능한 항목은 체크리스트(`- [ ]`) 사용
- 경로/명령어/환경변수는 백틱으로 표기 (`docs/api-spec.md`, `npm run build`)

### 4-3. 문서별 필수 섹션(권장)

- 목적
- 대상 독자
- 사용/운영 절차
- 점검 체크리스트
- 관련 문서 링크
- 마지막 업데이트 정보

---

## 5) 운영 프로세스 (핵심)

### 5-1. 문서 소유자(Owner) 지정

각 핵심 문서는 최소 1명의 담당자를 둡니다.

| 문서 | Owner | 검토 주기 | 마지막 검토일 |
|------|-------|-----------|----------------|
| `README.md` | TBD | 월 1회 | TBD |
| `docs/api-spec.md` | TBD | 배포 전/스키마 변경 시 | TBD |
| `docs/data-model.md` | TBD | DB 변경 시 | TBD |
| `docs/production-checklist.md` | TBD | 릴리즈 전 | TBD |

> `TBD`는 팀 담당자를 정한 뒤 즉시 채웁니다.

### 5-2. 변경 트리거 기준

아래 상황이 발생하면 관련 문서를 같은 작업 단위에서 함께 갱신합니다.

- API 라우트/응답 스키마 변경
- DB 스키마/마이그레이션 변경
- 인증/권한 흐름 변경
- 배포/환경변수 변경
- 사용자 플로우(UI) 변경

### 5-3. PR 체크리스트(권장)

PR 본문에 아래 항목을 포함해 문서 누락을 줄입니다.

- [ ] 이번 변경과 연관된 문서를 확인했다.
- [ ] 문서 업데이트가 필요하면 같은 PR에 반영했다.
- [ ] 업데이트가 불필요하다면 사유를 PR에 남겼다.

---

## 6) 작업 방식 템플릿

문서 작업 요청은 아래 포맷으로 작성하면 누락이 줄어듭니다.

```md
[문서 작업 요청]
- 목적:
- 범위(파일):
- 수정 항목:
- 제외 항목:
- 완료 기준(DoD):
```

예시:

```md
[문서 작업 요청]
- 목적: README만 보고 로컬 실행 가능하게 개선
- 범위(파일): README.md
- 수정 항목: 설치, 실행, 환경변수, 배포 명령
- 제외 항목: API 상세 명세
- 완료 기준(DoD): 신규 기여자가 10분 내 실행 성공
```

---

## 7) 월간 문서 점검 루틴 (30~45분)

- [ ] `README.md`의 설치/실행/환경변수 설명이 현재 코드와 일치하는지 확인
- [ ] `docs/api-spec.md`와 실제 API 응답 필드 차이 점검
- [ ] `docs/data-model.md`와 최신 마이그레이션 차이 점검
- [ ] 배포 체크리스트(`docs/production-checklist.md`)의 항목 유효성 점검
- [ ] 사용하지 않는 문서/깨진 링크/중복 설명 정리

---

## 8) 적용 우선안 (현재 프로젝트 권장)

당장 실무 효율을 높이려면 아래 순서를 권장합니다.

1. `README.md`를 온보딩 중심으로 먼저 최신화
2. `docs/api-spec.md`, `docs/data-model.md`를 코드/DB 기준으로 동기화
3. 공통 문체/섹션 규칙으로 전체 문서 톤 정리
4. Owner/검토 주기/PR 체크리스트를 루틴으로 고정

---

문서 관리는 "한 번 잘 쓰기"보다 "계속 맞춰가기"가 중요합니다. 이 가이드는 빠른 수정(D)부터 체계 운영(Owner/주기/체크리스트)까지 한 흐름으로 연결하는 것을 목표로 합니다.
